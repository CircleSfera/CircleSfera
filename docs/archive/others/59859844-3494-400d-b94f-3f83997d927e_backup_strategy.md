# Backup Strategy for CircleSfera

This document outlines the backup procedures for the CircleSfera backend, covering database, file storage, and cache persistence.

## 1. Database (PostgreSQL)

Since we are running PostgreSQL in a Docker container, we use `pg_dump` to create backups.

### Automated Backup Script

Create a script `backup_db.sh`:

```bash
#!/bin/bash
BACKUP_DIR="./backups/db"
mkdir -p $BACKUP_DIR
TIMESTAMP=$(date +"%Y%m%d_%H%M%S")
FILENAME="$BACKUP_DIR/db_backup_$TIMESTAMP.sql.gz"

# Dump the database from the running container
docker exec CircleSfera-Postgres pg_dump -U CircleSfera CircleSfera | gzip > $FILENAME

# Keep only the last 7 days of backups
find $BACKUP_DIR -type f -name "*.sql.gz" -mtime +7 -delete

echo "Backup created: $FILENAME"
```

### Scheduling

Add to crontab (`crontab -e`) to run daily at 3 AM:

```cron
0 3 * * * /path/to/circlesfera-backend/backup_db.sh >> /var/log/circlesfera_backup.log 2>&1
```

### Restoration

To restore a backup:

```bash
gunzip < ./backups/db/db_backup_20240101_120000.sql.gz | docker exec -i CircleSfera-Postgres psql -U CircleSfera -d CircleSfera
```

## 2. File Storage (Uploads)

### Local Storage

If running with `STORAGE_PROVIDER=local`, user uploads are stored in the `./d` directory (or wherever `local.provider.ts` saves them, currently `./uploads`).

**Strategy**:

- Include the `./uploads` directory in your server's filesystem snapshots or backup routine (e.g., using `rsync` to an external server/S3 bucket).
- Example: `rsync -avz ./uploads user@backup-server:/backups/circlesfera/uploads`

### Cloudinary

If running with `STORAGE_PROVIDER=cloudinary`, files are stored on Cloudinary.

- **Strategy**: Enable Cloudinary's automatic backup feature (available on paid plans) or use their API to sync assets to a secondary buckets (e.g., AWS S3).

## 3. Redis Persistence

We configured Redis with `--appendonly yes` in `docker-compose.yml`.
This ensures data durability across restarts using AOF (Append Only File).

**Strategy**:

- The data resides in the `redis_data` Docker volume.
- To backup, you can mount this volume to a temporary container and archive the `appendonly.aof` file, but typically Redis data is treated as transient/cache and re-populated from the DB if lost, unless used for critical queues.

## 4. Disaster Recovery Drill

Every 3 months, perform a dry run:

1. Spin up a fresh stack.
2. Restore the database from the latest backup.
3. Verify application functionality and data integrity.
