# Pay-Per-View (PPV) Monetization Plan

## Objetivo
Permitir que cualquier usuario (desde el minuto cero) pueda establecer un precio Pay-Per-View (PPV) en sus `Posts`, `Stories` o `Frames`. Los usuarios que quieran consumir ese contenido deberán pagar el precio establecido para desbloquearlo visualmente.

### 1. Base de Configuración (`.env.production`)
- Utilizar el archivo `.env.production` de la raíz como "Fuente de Verdad" para credenciales de DB, secretos JWT y claves de API (Brevo).
- Crear un `.env` local (o `docker.env`) adaptando:
  - `DB_HOST=postgres`
  - `CORS_ORIGIN="http://localhost:3000,http://localhost:8080,http://localhost:8081"`
  - `VITE_API_URL=http://localhost:3000` (para el build del frontend).

### 2. Orquestación (Root `docker-compose.yml`)
- Pasar `VITE_API_URL` como `build.args` para que el Frontend y Landing se compilen con la URL correcta del API local.
- Asegurar que el healthcheck de Postgres bloquee el inicio del Backend hasta que la DB esté lista.
- Mapear volúmenes para persistencia de datos (ya implementado para DB/Redis).

### 3. Sincronización de Base de Datos
- Crear un script de inicialización o documentar los comandos `docker exec` para correr `prisma migrate deploy` y `prisma seed` dentro del contenedor del backend.

## Cambios en la Base de Datos (Prisma)

### 1. Actualización del modelo `User`
Todo usuario debe tener la capacidad de cobrar. Usaremos Stripe Connect.
```prisma
model User {
  // ...
  stripeAccountId       String?   // Cuenta de Stripe Connect para recibir pagos
  isMonetizationEnabled Boolean   @default(true) // Switch global de monetización
}
```

### 2. Actualización de Modelos de Contenido (`Post` y `Story`)
Añadiremos los campos que dictan si un post/historia cobra portada. (`Frames` son un tipo de `Post` bajo `type = FRAME`, por lo que hereda estos campos mágicamente).
```prisma
model Post {
  // ...
  isPremium Boolean @default(false)
  price     Float?  // Precio en número (ej. 5.99)
  currency  String  @default("USD")
}

model Story {
  // ...
  isPremium Boolean @default(false)
  price     Float?
  currency  String  @default("USD")
}
```

### 3. [NUEVO] Modelo de Compras (`Purchase`)
Necesitamos registrar qué usuario ha comprado qué contenido para darle acceso perpetuo o temporal.
```prisma
model Purchase {
  id              String   @id @default(uuid())
  buyerId         String
  sellerId        String
  
  targetType      String   // "post", "story", "frame"
  targetId        String   // ID del contenido desbloqueado
  
  amount          Float
  currency        String   @default("USD")
  status          String   @default("COMPLETED") // "PENDING", "COMPLETED", "FAILED"
  stripeSessionId String?  @unique
  
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  buyer           User     @relation("BuyerPurchases", fields: [buyerId], references: [id])
  seller          User     @relation("SellerSales", fields: [sellerId], references: [id])

  @@unique([buyerId, targetId]) // Un usuario solo puede comprar un post una vez
  @@index([buyerId])
  @@index([sellerId])
  @@map("purchases")
}
```

## Lógica del Backend (Flujo de Stripe)
1. **Interceptor de Contenido:** Cuando alguien haga `GET /posts/:id`, verificaremos si `post.isPremium == true`. Si lo es, y el usuario actual *no es el autor* ni *tiene un record en la tabla Purchase*, retornamos un **objeto de Post bloqueado** (sin `mediaUrl` y con la descripción censurada) para que el frontend dibuje el candado/botón de compra.
2. **Endpoint de Pago:** `POST /payments/create-checkout-session` generará una ventana de pago de Stripe.
3. **Webhook:** Al completarse el pago, el Stripe Webhook actualizará el `Purchase` a `COMPLETED` y enviará una notificación In-App al Vendedor: *"¡Alguien ha comprado tu post!"*.

3. **Mocking Visual:** Programar un test manual creando un post con `price: 10` y verificar que bloquea correctamente los datos sensibles en los endpoints de Feed y Perfil.

# Priority 5: Monetization Analytics (Creator & Admin Dashboards)

## Objetivo
Diferenciar correctamente la analítica de Creadores (mis ventas y ganancias) frente a la analítica Administrativa Global (volumen total de la plataforma y comisiones cobradas).

## Cambios Propuestos
### 1. Panel de Creadores (`PaymentsService` & `PaymentsController`)
- **Endpoint:** `GET /payments/analytics/creator`
- Lógica de Agregación filtrando el modelo `Purchase` donde el `sellerId = req.user.id` y `status = 'COMPLETED'`.
- Devolveremos:
  - `totalEarnings` (80% del valor de las ventas históricas)
  - `earningsThisMonth` (80% de ventas del mes actual)
  - `recentSales` (Últimas ventas con detalles del comprador y contenido)

### 2. Panel Administrativo (`AdminService` & `AdminController`)
- **Endpoint:** `GET /admin/analytics/monetization`
- Lógica de Agregación leyendo **todas** las compras globales (`status = 'COMPLETED'`).
- Devolveremos:
  - `globalVolume` (Volumen BRUTO que ha pasado por la plataforma)
  - `platformRevenue` (El 20% retenido por CircleSfera mediante Stripe Application Fees)
  - `recentTransactions` (Historial global sin censura)

## Plan de Verificación
1. Obtener datos de creador simulando un seller.
2. Hacer request como Admin y verificar que `globalVolume` refleja el total de todos los sellers.

# Priority 6: Frontend PPV Integration

## Objetivo
Conectar el backend de Monetización Universal PPV con el Frontend de `circlesfera-frontend`. Esto incluye mostrar los candados de contenido, el botón de pago con Stripe, y los paneles de control de monetización (Creator y Admin).

## Cambios Propuestos
### 1. Interfaz de Feed y Perfiles (Consumidor)
- Cuando el post/historia llegue bloqueado (sin `mediaUrl` y con `isPremium: true`), renderizar un componente superpuesto: **"Contenido Exclusivo. Desbloquear por $X.XX"**.
- Al hacer click, redirigir al endpoint de backend `/payments/checkout` para abrir la sesión de Stripe.

### 2. Panel de Monetización de Creador (Creator Dashboard)
- Nueva sección de **"Monetización"** en los ajustes o perfil del usuario.
- Botón **"Conectar con Stripe"** si no tiene `stripeAccountId`. Llama a `/payments/connect-onboarding`.
- Mostrar gráficos o tarjetas de **Ingresos Totales** e **Ingresos del Mes** consumiendo `/payments/analytics/creator`.

### 3. Panel de Administrador (Admin Dashboard)
- En el dashboard global, añadir tarjetas de **Ganancias de CircleSfera (20%)** consumiendo `/admin/analytics/monetization`.

## Plan de Verificación

2. Crear un Post PPV desde la cuenta A.
3. Iniciar sesión con la cuenta B, comprobar si el Post se muestra bloqueado y si el botón redirige a Stripe de prueba.

# Priority 7: Verification Levels & Creator Roles

## Objetivo
Implementar una taxonomía de usuarios para diferenciar entre personas físicas, empresas y creadores profesionales, con diferentes niveles de verificación y badges asociados.

## Cambios Propuestos

### 1. Backend (Prisma & Enums)
- Añadir `VerificationLevel` (`BASIC`, `VERIFIED`, `BUSINESS`, `ELITE`).
- Añadir `AccountType` (`PERSONAL`, `CREATOR`, `BUSINESS`).
- Actualizar el modelo `User` para incluir estos campos.

### 2. Frontend (UI/UX)
- Mostrar badges de verificación junto al nombre del usuario en el Perfil y en el Feed.
- `BASIC`: Sin badge.
- `VERIFIED`: Badge azul (Standard Verification).
- `ELITE`: Badge dorado (Top Creators).
- `BUSINESS`: Badge gris/maletín (Business Accounts).

### 3. Business Logic
- `Minuto 0`: Todos los niveles pueden monetizar (PPV).
- `Verification Request`: Endpoint para que un usuario solicite subir de nivel subiendo ID/Documentación (Futuro).

## Plan de Verificación
1. Migrar base de datos.
2. Actualizar tipos en el Frontend.
3. Inyectar badges en `PostCard` y `ProfileHeader`.

# Priority 8: Type-Safety & Code Quality Fixes

## Objetivo
Resolver todos los errores de linting ("unsafe any") y las inconsistencias de tipos de Prisma detectadas en el Panel Admin y servicios asociados para mantener un estándar profesional.

## Cambios Propuestos

### 1. Sincronización de Prisma (Backend)
- Asegurar que `verificationLevel` y `accountType` sean reconocidos nativamente por el Prisma Client.
- Eliminar el uso de `any` en `AdminService` y `AdminController`.
- Usar los Enums generados `VerificationLevel` y `AccountType` de `@prisma/client`.

### 2. Tipado en Frontend
- Refinar la interfaz `AdminUser` e `IUser` para incluir los nuevos campos.
- Reemplazar `(users as any[]).map` por un tipado explícito en `UserVerificationTab.tsx`.

## Plan de Verificación
1. Eliminar `as any` en `AdminService`.
2. Verificar que el compilador de TypeScript (TS Server) no lance errores.
3. Ejecutar `npm run build` en el backend para validar el tipado.
